# Design patterns — Creational

Creational patterns manage **object creation** so code stays flexible when families of products, construction steps, or uniqueness constraints evolve.

| Pattern | One-line idea | Official |
|--------|----------------|----------|
| Factory Method | Subclasses choose which concrete type to create | [Factory Method](https://refactoring.guru/design-patterns/factory-method) |
| Abstract Factory | Create families of related objects without naming concrete classes | [Abstract Factory](https://refactoring.guru/design-patterns/abstract-factory) |
| Builder | Step-by-step assembly of a complex object | [Builder](https://refactoring.guru/design-patterns/builder) |
| Prototype | Clone a configured instance instead of building from scratch | [Prototype](https://refactoring.guru/design-patterns/prototype) |
| Singleton | One shared instance—use sparingly; testability and globals cost | [Singleton](https://refactoring.guru/design-patterns/singleton) |
