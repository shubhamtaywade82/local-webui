# Design patterns — Structural

Structural patterns explain how to **compose** classes and objects for new capabilities: adapters for legacy interfaces, decorators for layered behavior, facades to simplify subsystems, proxies for lazy or controlled access, etc.

| Pattern | One-line idea | Official |
|--------|----------------|----------|
| Adapter | Wrap an incompatible API behind the interface callers expect | [Adapter](https://refactoring.guru/design-patterns/adapter) |
| Bridge | Split abstraction from implementation so both vary independently | [Bridge](https://refactoring.guru/design-patterns/bridge) |
| Composite | Tree structures where leaves and containers share one type | [Composite](https://refactoring.guru/design-patterns/composite) |
| Decorator | Add responsibilities by wrapping, without subclass explosion | [Decorator](https://refactoring.guru/design-patterns/decorator) |
| Facade | One entry point hides a messy subsystem | [Facade](https://refactoring.guru/design-patterns/facade) |
| Flyweight | Share immutable intrinsic state across many instances | [Flyweight](https://refactoring.guru/design-patterns/flyweight) |
| Proxy | Stand-in controls access, cost, or lifecycle of another object | [Proxy](https://refactoring.guru/design-patterns/proxy) |
