# Design patterns — Behavioral

Behavioral patterns organize **communication and responsibility**: algorithms that can vary (strategy), steps in a skeleton method (template method), notifications (observer), encapsulated requests (command), object graphs with controlled traversal (iterator/visitor), etc.

| Pattern | One-line idea | Official |
|--------|----------------|----------|
| Chain of Responsibility | Pass a request along a chain until something handles it | [Chain of Responsibility](https://refactoring.guru/design-patterns/chain-of-responsibility) |
| Command | Encapsulate an action as an object (undo, queue, logging) | [Command](https://refactoring.guru/design-patterns/command) |
| Iterator | Sequential access without exposing internals | [Iterator](https://refactoring.guru/design-patterns/iterator) |
| Mediator | Central hub reduces many-to-many coupling | [Mediator](https://refactoring.guru/design-patterns/mediator) |
| Memento | Snapshot state for rollback without breaking encapsulation | [Memento](https://refactoring.guru/design-patterns/memento) |
| Observer | Subscribers react to subject changes | [Observer](https://refactoring.guru/design-patterns/observer) |
| State | Behavior changes when an object’s internal state changes | [State](https://refactoring.guru/design-patterns/state) |
| Strategy | Swap algorithms behind one interface | [Strategy](https://refactoring.guru/design-patterns/strategy) |
| Template Method | Base class defines skeleton; subclasses fill hooks | [Template Method](https://refactoring.guru/design-patterns/template-method) |
| Visitor | Add operations across a type hierarchy without changing each class | [Visitor](https://refactoring.guru/design-patterns/visitor) |

**Catalog:** [Behavioral patterns](https://refactoring.guru/design-patterns/behavioral-patterns)
